## 1️⃣ 为什么需要 SDD

### 1.1 AI 需求的痛点

```
❌ 模糊指令："做个用户登录功能"
✅ AI 输出：没验证码、没密码加密、没失败限制...

问题根源：不是 AI 的问题，是人的问题——需求未明确规范
```

> 🎯 **核心结论**：SDD 把"模糊需求"变成"明确规范"，让 AGI 按规则干活，从根源上减少返工、提升质量、降低维护成本。

### 1.2 Vibe Coding（氛围编程）的陷阱

| 概念 | 说明 |
|------|------|
| **提出者** | Andrej Karpathy（2025年2月） |
| **核心** | 基于 LLM 的自然语言驱动开发，"感觉对了就行" |
| **风险** | 🍎 糖霜苹果：表面光鲜，内里空心 |
| **问题** | "感觉"因人而异：新手=能跑就行，资深=性能优先，AI=凑逻辑就行 |

> 🔑 **关键洞察**：  
> AGI 时代，AI 能帮我们干活，但如果没有 SDD，AI 生成的代码只会"随机肆意发挥"，就如同 AI 幻觉。

## 2️⃣ SDD 到底是什么

### 2.1 本质定义

```
SDD ≠ 纯写文档
SDD = 和 AI 说清人话 + 定好规矩 + 达成共识
```

> SDD 是"和 AI 合作的沟通手册"，用固定格式把"要做什么、做成什么样、怎么一步步做"说清楚。

### 2.2 三大核心阶段

#### 📋 阶段一：需求阶段（requirements.md）
```markdown
## 用户故事
作为 [角色]，我想要 [功能]，以便 [价值]
示例：作为普通用户，我想通过手机号+验证码登录，这样不用记密码，登录更方便

## 验收标准（WHEN...THEN...）
- WHEN 用户输入错误密码3次 THEN 锁定账号10分钟
- WHEN 验证码过期 THEN 提示"验证码失效"
- WHEN 同一手机号1分钟内请求 THEN 限制获取频次
```

#### 🏗️ 阶段二：设计阶段（design.md）
```markdown
## 架构流程
前端 → 网关 → 登录服务 → 数据库

## 数据模型
用户表：手机号、验证码、创建时间、错误次数...

## 技术选型
- 验证码存储：Redis（自动过期）
- 密码加密：BCrypt
```

#### ✅ 阶段三：任务阶段（tasks.md）
```markdown
- [ ] 任务1：设计用户表结构
- [ ] 任务2：开发"获取验证码"接口
- [ ] 任务3：开发前端登录页面
- [ ] 任务4：联调接口并测试验收标准
```

> ⚠️ **重要原则**：SDD 文档要"活"，与代码同步更新，确保"说的"和"做的"始终一致。

### 2.3 SDD 三大境界（成熟度等级）

| 境界 | 名称 | 核心特征 | 适用阶段 |
|------|------|----------|----------|
| 🔹 L1 | **Spec-first**<br>规格优先 | 先写规格→生成代码→用完可丢 | ✅ 当前主流，入门首选 |
| 🔹 L2 | **Spec-anchored**<br>规格锚定 | 规格长期维护+版本控制，代码跟着规格走 | 🎯 推荐目标，团队协作 |
| 🔹 L3 | **Spec-as-source**<br>规格即源码 | 人只写规格，代码100%自动生成，规格=真相源 | 🔮 理想状态，当前难实现 |

> 📌 **现实判断**：当前行业大多处于 L1-L2 之间，L3 对文档质量与 AI 能力要求极高。

## 3️⃣ SDD 工具核心概念

### 3.1 EARS 模式（需求语法简易方法）

```
EARS = Easy Approach to Requirements Syntax
提出：Alistair Mavin & Rolls-Royce 团队（2009年）
应用：空客、NASA、博世、英特尔、西门子等
```

#### 📐 标准格式
```
虽然 <可选前置条件>，
但当 <可选触发条件> 时，
<系统名称> 应 <系统响应>
```

#### 💡 实战示例
```markdown
## 错误处理需求
WHEN 用户输入错误的账号或者密码 3 次
THEN 系统应该锁定该账号 5 分钟

## 验证码需求
WHEN 用户请求获取验证码
AND 该手机号 1 分钟内已获取过
THEN 系统应返回"请求过于频繁，请稍后再试"
```

#### ✅ EARS 三大优势
| 优势 | 说明 |
|------|------|
| 🔹 倒逼需求清晰化 | 杜绝"大概""可能"等模糊描述，强制明确 场景→行为→结果 |
| 🔹 提升可测性 | 触发条件+预期结果=天然测试用例，便于自动化验证 |
| 🔹 降低 AI 理解成本 | 结构化格式减少 AI 猜测，生成代码更精准 |

## 4️⃣ 理论与思考

### 4.1 Vibe Coding vs SDD 对比

| 维度 | Vibe Coding | SDD |
|------|-------------|-----|
| 🎯 适用人群 | 非编程人员、快速原型 | 专业团队、生产级系统 |
| ⚡ 开发速度 | 快（但返工风险高） | 前期慢，后期稳 |
| 🔧 工程质量 | 难保证，依赖个人感觉 | 规范约束，可验证可追溯 |
| 🔄 系统维护 | 难（代码与意图脱节） | 易（规格即文档，同步演进） |

### 4.2 SDD 不是万能药：使用建议

```
✅ 适合场景：
• 大型系统/复杂业务规则
• 多团队协作/跨部门沟通
• 长期维护/高可靠性要求
• AI 深度参与的代码生成

❌ 慎用场景：
• 简单小工具/一次性脚本
• 快速验证想法的 MVP
• 个人学习/实验性项目

💡 落地策略：
• 增量式引入：只在代码变更区域补充规范
• 绞杀者模式：老系统逐步替换，新模块按 SDD 开发
• 细粒度拆分：避免一次性写完整系统规范，降低 AI 上下文负担
```

## 5️⃣ 写在最后

### 🎁 核心金句摘录

> 🔹 "想，都是问题；做，才是答案。现在动手，说干就干的执行力，永远比原地徘徊更有意义。"

> 🔹 "SDD 不是要消除 AI 的不确定性，而是通过规范对齐，将概率性输出引导至可预测、可验证、可干预的轨道。"

> 🔹 "规格即真相来源——避免出现'文档和代码对不上'的终极难题。"

> 🔹 "知行合一：知是行之始，行是知之成。只有实际动手实践，才能深刻明白 SDD 背后的哲学。"

### 🚀 实践 Checklist

```markdown
## 起步清单

### 第一步：选对场景
- [ ] 选择规则清晰、高频重复、与核心业务解耦的模块试点
- [ ] 避免从最复杂的核心系统开始

### 第二步：建立模板
- [ ] 创建 requirements.md / design.md / tasks.md 模板
- [ ] 采用 EARS 格式编写验收标准
- [ ] 明确"用户故事+验收标准+技术设计+任务拆分"四要素

### 第三步：配置协作
- [ ] 用 CLAUDE.md / AGENTS.md 定义 AI 角色边界
- [ ] 设置人工 Review 节点，关键决策人类把关
- [ ] 建立"Spec→Code→Test→Update Spec"闭环流程

### 第四步：持续迭代
- [ ] 代码变更时同步更新对应 Spec 文档
- [ ] 定期 Review 规格与实现的一致性
- [ ] 收集反馈，优化模板与协作流程
```
